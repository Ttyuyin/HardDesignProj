2. Score Comparison (Before vs After)
================================================================================

  Test Sample: "Hello! 日本語プログラミングテスト。データ構造。"
  Encoding: Shift-JIS (cp932), 47 bytes

  Before (raw agent scores):
    GBK:           0.9222
    Shift-JIS:     0.6296
    After softmax: GBK(0.9486) >> Shift-JIS(0.0509)
    Result: MISDETECTED as GBK

  After (raw agent scores):
    GBK:           0.6612  (reduced by kana penalty: 0.4815 * 0.65 = 0.313, capped 0.35)
    Shift-JIS:     0.6296  (unchanged)
    After softmax: Shift-JIS(0.5478) > GBK(0.4469)
    Result: CORRECTLY detected as Shift-JIS

  Key insight: The Shift-JIS bytes, when decoded as GBK, produce CJK mojibake
  with no kana characters. But when decoded as cp932, they reveal strong kana
  content (kana_ratio = 0.4815). This cross-encoding check is the fix.

5. Final Recommendations
================================================================================

  [PASS] Shift-JIS detection is now correct for texts with kana content.
         Short pure-CJK SJIS texts (no kana, e.g. "日本語") remain ambiguous
         — this is a fundamental limitation of byte-level CJK detection.

  [INFO] GB2312 → GBK mapping is expected behavior and not a bug.
         The detector's DETECTION_ORDER includes GB2312 but there is no
         separate GB2312 agent. GB2312 content is valid GBK.

  [INFO] Cross-encoding overlap (GBK/Big5/Shift-JIS share byte ranges)
         is a known limitation. The negative_features mechanism provides
         a configurable framework to add disambiguation rules.

  [SUGGEST] Future work could extend negative_features to:
    - Big5 agent: cp932 kana check (for Big5 vs Shift-JIS disambiguation)
    - UTF-8 agent: GBK/Big5 overlap penalty
    - More negative features per agent: "cjk", "bopomofo", "hangul"
